# GPU: externally hosted, GPU-backed inference runtimes

The `GPU` custom resource leases a GPU from an external provider (**RunPod** today,
pluggable for others) and runs your container image on it as an inference runtime
reachable over HTTP. The Kubernetes cluster is the **control plane only** - GPU
hardware, drivers, scheduling and inference execution stay with the provider.

A `GPU` is **not** a worker node, a device, an `nvidia.com/gpu` request, or a
replacement for the NVIDIA device plugin / Dynamic Resource Allocation.

```text
GPU CR -> RunPod API -> GPU instance runs your image -> HTTP endpoint ready
       -> connection Secret (url + token) -> Inference Gateway
```

## This example: Ornith-1.0 on RunPod, served by vLLM

[`gpu.yaml`](./gpu.yaml) leases one 80GB GPU on RunPod and serves
[**Ornith-1.0**](https://huggingface.co/deepreinforce-ai) - an agentic-coding model -
through vLLM's OpenAI-compatible API. vLLM pulls the weights from Hugging Face by id
at startup, so **nothing has to be baked into the image or mounted**:

```yaml
spec:
  provider: runpod
  image: vllm/vllm-openai:latest
  command: # appended to the image's vLLM entrypoint
    - "--model"
    - "deepreinforce-ai/Ornith-1.0-9B"
    - "--served-model-name"
    - "Ornith-1.0"
    - "--host"
    - "0.0.0.0"
    - "--port"
    - "8000"
    - "--max-model-len"
    - "32768"
  gpuTypes:
    - NVIDIA A100 80GB PCIe
    - NVIDIA H100 80GB HBM3
  endpoint:
    port: 8000
    readinessPath: /health
  maxRuntime: 2h
```

The RunPod driver leases a **single** GPU, so this example uses the dense **9B**
checkpoint, which fits one 80GB card. The `35B` / `397B` MoE checkpoints need
multi-GPU sharding and are out of scope for this first cut.

## Apply

```bash
# Put your RunPod API key into the Secret (spec.credentialsRef) first.
kubectl apply -f gpu.yaml
kubectl -n inference-gateway get gpu ornith -w
```

```text
NAME     PHASE   URL                                    INSTANCE   AGE
ornith   Ready   https://abc123-8000.proxy.runpod.net   abc123     4m
```

`Ready=True` is reported **only** after `endpoint.readinessPath` answers with a 2xx -
the provider reporting "running" is not enough, since the weights still have to
download and load first.

## Lifecycle

```text
Pending -> Provisioning -> Starting -> Ready
                                    -> Terminating -> Expired
(any step) -> Failed
```

- The RunPod allocation is tagged with the GPU object UID, so reconciliation
  (including after a controller crash) recovers the existing allocation instead of
  creating a duplicate.
- `maxRuntime` is **required**. When it elapses the allocation is released and the
  resource is parked in `Expired` - it is **not** reprovisioned automatically.
  Recreate it or change a spec field to lease again.
- A finalizer releases the allocation on deletion. Provider `Not Found` is treated
  as successful cleanup.

## Connection Secret

Once the runtime is ready the controller creates/updates `ornith-connection`:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: ornith-connection
stringData:
  url: https://abc123-8000.proxy.runpod.net
  apiKey: <per-allocation token generated by the operator>
```

The `apiKey` is a **per-allocation token generated by the operator**, never the
RunPod management key (which never leaves the reconcile loop). The operator injects
this token into the runtime as the `API_KEY` env var.

> **Auth note:** vLLM enforces a key via `--api-key` / the `VLLM_API_KEY` env var,
> not `API_KEY`, so this demo serves openly and relies on the unguessable RunPod
> proxy URL as the boundary. To make vLLM _require_ the token, surface it to the
> runtime as `VLLM_API_KEY` (this per-image env wiring is a documented follow-up).

## Consuming it from a Gateway

Auto-discovery (an optional `gpuRef`) is out of scope for this first cut. A Gateway
consumes the connection Secret through the standard provider `env` / `secretKeyRef`
mechanism - see the `Gateway` in [`gpu.yaml`](./gpu.yaml):

```yaml
providers:
  - name: ornith
    enabled: true
    env:
      - name: LLM_BASE_URL
        valueFrom:
          secretKeyRef:
            name: ornith-connection
            key: url
      - name: LLM_API_KEY
        valueFrom:
          secretKeyRef:
            name: ornith-connection
            key: apiKey
```
